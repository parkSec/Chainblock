# 대학생 코딩 버그바운티 플랫폼 — 통합 문서 (PRD + SRS + 화면설계)

버전: 1.1  
작성일: 2026-06-01

요약:
- 백엔드(서버)가 없는 100% 온체인/탈중앙화 DApp (Serverless Architecture)
- 대학생들이 코딩 중 발생하는 버그나 문제를 올리고, 다른 학생이 이를 해결해주는 C2C(Student-to-Student) 버그바운티 플랫폼
- 질문자는 코인을 지불하여 답변(정답 코드)을 열람할 수 있음
- 스마트 컨트랙트를 통해 답변 열람 시 코인이 답변자에게 자동 전송됨
- Web3 지갑(MetaMask 등)을 이용한 로그인, 결제 및 데이터 서명/암호화
- 질문 작성, 코드 하이라이팅 지원, 답변 작성 및 가리기(비대칭 암호화 기반 블라인드) 기능

## 1. 목표 및 배경 (PRD)
- **목적:** 대학생들이 코딩 과제나 개인 프로젝트 중 겪는 오류를 블록체인 기반의 보상 시스템을 통해 빠르고 효율적으로 해결할 수 있도록 돕는 지식 공유 플랫폼 구축.
- **배경:** 중앙화된 서버 없이(백엔드리스) 순수 Web3 기술만으로 투명하게 운영되는 DApp을 구축.
- **핵심 가치:** 
  - 질문자: 소정의 코인을 지불하여 확실한 문제 해결(디버깅) 방법 획득.
  - 답변자: 자신의 코딩 실력을 발휘하여 문제를 풀고, 그에 대한 합당한 보상(코인) 획득.

## 2. 기능 목록 (High-level)
- **지갑 연동:** MetaMask 등을 통한 Web3 로그인 (사용자 식별 및 지갑 주소 확보).
- **질문(버그) 등록:** 에러 내용, 작성하던 코드, 걸어둘 현상금(Bounty 코인 양)을 IPFS에 저장 후, CID와 보상금을 스마트 컨트랙트에 등록.
- **질문 목록/조회:** 블록체인에서 질문 목록을 불러오고 상세 내용은 IPFS를 통해 렌더링.
- **답변 등록:** 코드를 본 다른 대학생이 정답을 등록. 중앙 서버가 없으므로 정답 코드는 질문자의 공개키(Public Key)로 비대칭 암호화하여 IPFS에 저장 (블라인드 처리).
- **정답 열람 및 보상 지급:** 질문자가 특정 답변을 선택해 코인을 컨트랙트로 전송하면, 스마트 컨트랙트 상에 열람 권한이 기록되고, 프론트엔드에서 자신의 개인키(Private Key)로 정답을 복호화하여 열람. 코인은 답변자에게 전송됨.

## 3. 상세 요구사항 (SRS)

### 3.1 회원 및 권한
- 중앙화된 DB가 없으므로 별도 회원가입 없이 지갑 연결 상태로만 작동한다.
- 닉네임 등의 기본 정보는 필요시 컨트랙트 또는 분산 저장소에 기록한다.

### 3.2 질문자(Questioner) 기능
- **질문 작성:** 제목, 본문, 에러 메시지, 현재 코드를 입력하여 트랜잭션과 함께 배포한다.
- **바운티 설정:** 질문을 등록할 때 지불할 코인의 양을 명시한다.
- **답변 목록 확인:** 컨트랙트에 등록된 답변의 '해설 요약(스포일러 방지, 평문 저장)'만 볼 수 있다.
- **답변 채택/열람:** 코인을 지불(트랜잭션 발생)하면, 프론트엔드가 암호화된 정답 코드를 지갑을 통해 복호화(Decrypt)하여 보여준다.

### 3.3 답변자(Solver) 기능
- **문제 탐색:** 프론트엔드에서 컨트랙트를 폴링(Polling)하거나 이벤트를 구독하여 문제 목록을 확인.
- **답변 작성:** 질문자의 공개키를 이용해 정답 코드를 암호화하여 제출(트랜잭션).
- **보상 수령:** 질문자가 답변 열람 트랜잭션을 실행하면, 컨트랙트에 의해 자동으로 지갑으로 코인이 전송된다.

## 4. 스마트컨트랙트 및 토크노믹스 명세

- **네트워크:** Ethereum Sepolia Testnet
- **사용 토큰:** Sepolia ETH (또는 컨트랙트로 배포한 자체 ERC-20 토큰).
- **주요 함수 (스마트 컨트랙트):**
  - `createQuestion(string ipfsCID, uint256 bountyAmount)`: 질문 등록 (선택적으로 보상금 예치 가능).
  - `submitAnswer(uint256 questionId, string encryptedAnswerCID, string summary)`: 답변자 답변 등록.
  - `unlockAnswer(uint256 questionId, uint256 answerId)`: 질문자가 payable로 호출하여 코인을 지불하고 열람 권한 획득. 금액은 답변자에게 전송됨.
  - `isUnlocked(uint256 questionId, uint256 answerId) returns (bool)`: 특정 답변이 결제되었는지 프론트엔드에서 확인하는 읽기 전용 함수.

## 5. 데이터 모델 (온체인 & IPFS)

백엔드 DB를 없애고 모든 데이터를 탈중앙 환경에 저장한다.

### 5.1 온체인(On-chain) 데이터 (Smart Contract State)
가스비 최적화를 위해 최소한의 메타데이터와 상태값만 저장:
- **Question:** ID, 질문자 주소(Author), 보상금(Bounty), IPFS CID (질문 원문)
- **Answer:** ID, Question ID, 답변자 주소(Solver), 해설 요약(Summary), 암호화된 정답의 IPFS CID, 열람 여부(Unlocked Status)

### 5.2 오프체인 분산 저장소 (IPFS / Pinata 등)
무거운 텍스트 데이터 보관:
- **질문 JSON:** { "title": "...", "content": "...", "code": "..." }
- **답변 JSON:** { "encryptedCode": "0x... (질문자의 공개키로 암호화된 문자열)" }

## 6. DApp 아키텍처 및 스택 제안 (Serverless)

- **Frontend:** Next.js (React) 또는 Vite (React)
- **Styling:** Tailwind CSS
- **Web3 연동:** ethers.js / viem / wagmi
- **암복호화 로직:** MetaMask의 `eth_getEncryptionPublicKey` 및 `eth_decrypt` RPC 메서드 활용, 또는 Lit Protocol 도입.
- **Smart Contract:** Solidity, Hardhat/Foundry, Sepolia Testnet 배포
- **Storage:** IPFS (Pinata 서비스 활용 등으로 프론트엔드에서 직접 업로드)
- **Hosting:** Vercel 또는 GitHub Pages (정적 웹 호스팅)

## 8. 수용 기준(AC) 및 테스트 시나리오

1. **지갑 연결 및 공개키 획득 테스트:** MetaMask 연결 후 질문 작성 시 지갑으로부터 암호화에 필요한 공개키(Public Key)를 정상적으로 추출할 수 있는가?
2. **IPFS 업로드 테스트:** 프론트엔드에서 백엔드 없이 IPFS(Pinata 등)로 JSON 데이터를 업로드하고 CID를 반환받을 수 있는가?
3. **온체인 상태 반영:** 질문 등록 시 트랜잭션이 성공적으로 Sepolia에 기록되고, 메인 화면에서 해당 내역이 즉시 폴링/업데이트 되는가?
4. **비대칭 암호화 블라인드 테스트 (Serverless 보안):**
   - 답변자가 코드를 제출할 때 질문자의 공개키를 사용하여 데이터가 성공적으로 암호화(Encrypt)되어 IPFS에 저장되는가?
   - 중간에서 누군가 IPFS CID를 알아내더라도 암호화되어 있어 원본 코드를 볼 수 없는가?
5. **결제 및 복호화(Decrypt) 테스트:**
   - 질문자가 `열람하기` 버튼을 누르면 컨트랙트로 코인이 지불되는가?
   - 트랜잭션 완료 확인 후, 프론트엔드에서 MetaMask의 `eth_decrypt`를 호출하여 질문자 본인의 개인키로만 원본 코드가 복호화되어 화면에 렌더링되는가?
   - 답변자 지갑 주소로 코인 송금이 정상적으로 이루어졌는가?

## 9. 개발 페이즈 및 일정 (1 Man-Day 단위 분할)

1인 개발자(1 Man-Day, 하루 8시간 기준)가 하루에 소화할 수 있는 분량으로 아주 잘게 쪼갠 **8일(8 Phases) 마일스톤**입니다.

### Day 1: 프로젝트 셋업 및 컨트랙트 기초 설계
- **목표:** 개발 환경 구축 및 스마트 컨트랙트 뼈대(Struct, State) 작성
- **주요 태스크:**
  1. 프론트엔드 레포지토리(Next.js) 및 컨트랙트 폴더(Hardhat) 초기화
  2. `Question`, `Answer` 등 핵심 온체인 데이터 구조체(Struct) 설계
  3. `BugBounty.sol` 기본 상태 변수 및 이벤트(Events) 정의
  4. 간단한 배포 스크립트 작성 및 로컬 네트워크(Hardhat Network) 구동 확인

### Day 2: 스마트 컨트랙트 핵심 로직 및 테스트
- **목표:** 컨트랙트의 모든 함수 구현 완료 및 단위 테스트
- **주요 태스크:**
  1. `createQuestion`, `submitAnswer`, `unlockAnswer` 함수 구현
  2. 함수별 `require` 조건(보상금 검증, 권한 검증 등) 및 보안 로직 추가
  3. 로컬 네트워크에서 Hardhat/Chai를 이용해 주요 시나리오 단위 테스트(Unit Test) 작성 및 통과
  4. Sepolia Testnet 배포 및 프론트엔드용 ABI 추출

### Day 3: 프론트엔드 뼈대 & 지갑 연동
- **목표:** UI 기본 틀 마련 및 MetaMask 연동
- **주요 태스크:**
  1. Tailwind CSS 세팅 및 메인 화면/상세 화면 레이아웃 퍼블리싱 (껍데기)
  2. Wagmi/Viem (또는 ethers.js) 설치 및 Web3Provider 설정
  3. 헤더에 '지갑 연결(Connect Wallet)' 버튼 구현 및 상태 관리
  4. 연결된 지갑 주소, 네트워크(Sepolia) 정보 UI에 표시

### Day 4: IPFS 연동 로직 (Pinata)
- **목표:** 백엔드 DB를 대신할 IPFS 읽기/쓰기 기능 구현
- **주요 태스크:**
  1. Pinata API Key 발급 및 환경 변수(.env) 설정
  2. 프론트엔드 내에 IPFS 업로드 유틸리티 함수(`uploadJSONToIPFS`) 작성
  3. IPFS 다운로드 유틸리티 함수(`fetchFromIPFS`) 작성
  4. 로컬 상태에서 가상의 JSON 데이터를 IPFS에 올리고 CID로 다시 불러오는 테스트 페이지 구현

### Day 5: 온체인 데이터 읽기 & UI 렌더링
- **목표:** 컨트랙트에 저장된 질문/답변 목록을 화면에 그리기
- **주요 태스크:**
  1. `BugBounty.sol` 컨트랙트 인스턴스를 프론트엔드에 연결
  2. 메인 페이지: 컨트랙트에서 모든 질문(Question) 목록 가져오기 로직 구현
  3. 각 질문의 CID를 이용해 IPFS에서 제목/본문을 가져와 리스트 형태로 렌더링
  4. 질문 상세 페이지: 해당 질문에 달린 답변(Answer) 목록(요약만) 렌더링

### Day 6: 온체인 트랜잭션 쓰기 (질문/답변 등록)
- **목표:** 사용자가 직접 질문과 답변을 등록하는 기능 연결 (암호화 제외)
- **주요 태스크:**
  1. '질문 작성 폼' 구현 및 폼 제출 시 IPFS 업로드 -> `createQuestion` 트랜잭션 실행 연결
  2. '답변 작성 폼' 구현 및 폼 제출 시 IPFS 업로드 -> `submitAnswer` 트랜잭션 실행 연결
  3. 트랜잭션 진행 상태(Pending, Success, Revert)에 따른 로딩 스피너 및 알림(Toast) UI 적용

### Day 7: 비대칭 암호화(블라인드) 로직 구현
- **목표:** MetaMask의 암/복호화 API를 적용하여 정답 숨기기
- **주요 태스크:**
  1. 질문 등록 시, 작성자의 공개키를 `eth_getEncryptionPublicKey`로 가져와 컨트랙트나 IPFS 메타데이터에 함께 저장하도록 수정
  2. 답변자가 답변 제출 시, 해당 공개키를 이용해 정답 코드를 암호화(Encrypt)하여 IPFS에 저장하도록 로직 수정
  3. 암호화 유틸리티 함수 작성 및 테스트

### Day 8: 결제 후 복호화 열람 & 최종 배포
- **목표:** 코인 지불 후 정답을 확인하는 핵심 플로우 통합 및 라이브 배포
- **주요 태스크:**
  1. 질문 상세 페이지에 '열람하기(결제)' 버튼 추가 및 `unlockAnswer` (payable) 트랜잭션 연결
  2. 결제 완료 상태(Unlocked) 감지 후, MetaMask `eth_decrypt`를 호출하여 암호화된 코드 복호화(Decrypt) 적용
  3. 복호화된 원본 코드를 코드 블럭(Monaco Editor 등)에 예쁘게 렌더링
  4. 전체 통합 테스트(질문->답변->암호화->결제->복호화) 및 Vercel에 프론트엔드 정적 호스팅 배포
